## Per-axis classification

| Axis | Verdict | Why |
|---|---|---|
| **Auth** | **NATIVE** | `Authorization: Bearer xoxb-...` ‚Äî exact fit for refract's `Bearer` strategy. (Token-as-POST-param is a documented fallback, irrelevant to the native path.) |
| **Pagination** | **CUSTOM / near-miss** | Cursor lives at `response_metadata.next_cursor` ‚Äî a **nested/dotted path**, not a flat field. Spec ¬ß5's only shown config is `items_field: result` (flat). ¬ß6 says `Cursor (envelope next-cursor field)` but never shows a dotted `cursor_field:`. If `Cursor`'s field config only accepts flat names, this is a **GAP**; if it's generalized to accept a dotted path (`response_metadata.next_cursor`), it's native. Frozen spec text does not confirm either way ‚Äî flagged as the pagination near-miss. Termination is also three-way ambiguous ("empty, null, or non-existent") ‚Äî a real robustness nuance any generic Cursor implementation must handle. |
| **Request body** | **NATIVE (JSON path)**, GAP on the side | Slack accepts both `application/x-www-form-urlencoded` and JSON on the *same* method ‚Äî a generated client can just pick `TypedModel`+JSON and it's a clean fit. But `conversations.invite`'s `users` field is documented as "a comma separated list of user IDs" for form encoding ‚Äî a `list<string>` that must serialize as CSV, not JSON array, when the form path is chosen. Neither `TypedModel` nor `Assembled` in ¬ß7 describes per-field CSV serialization for form encoding ‚Äî a **near-miss** only if you pick the form path; sidestepped by choosing JSON. |
| **Responses & errors** | **GAP (headline)** | Every method ‚Äî including all 11 sampled ‚Äî returns **HTTP 200** for both success and business-logic failure; `{"ok":false,"error":"channel_not_found"}` etc. never touches a 4xx/5xx status. refract's `responses:` map (¬ß5, ¬ß9) is keyed by *HTTP status* ‚Üí one model each; it has no construct for "one status, two bodies, discriminated by an in-body flag." This is not expressible without a new concept (e.g. a `BodyFlag` error strategy keyed on `ok`/`error`). **429 is the one real exception** ‚Äî genuine non-2xx with `Retry-After` ‚Äî but it's cross-cutting across ~200+ methods with no "default responses for a resource" inheritance shown in ¬ß4/¬ß5, so it'd be hand-copied into every op file (DRY near-miss). |
| **`ReturnsError` test** | **Vacuous, exactly as suspected** | It's auto-generated by stubbing a 4xx/5xx. Slack never returns 4xx/5xx for `invalid_auth` / `channel_not_found` / `name_taken` / etc., so the generated test exercises a code path Slack never triggers, while the *actual* error path (200+`ok:false`) is silently untested by the default suite. Worse: with a naive status-based dispatcher, a real `200 {"ok":false,"error":"..."}` response gets parsed as the *success* model ‚Äî either a validation crash (required success fields missing) or silent misparse (fields all optional) ‚Äî and no auto test catches it. |
| **Operation/path shape** | **NATIVE, better than expected** | All calls hit a flat RPC endpoint `POST/GET slack.com/api/<method.name>` with zero path templating ‚Äî Slack's own `domain.method` dot-namespacing (`conversations.list`, `chat.postMessage`) already isomorphically matches refract's `<resource>/<operation>` directory nesting (¬ß1). `path:` degenerates to a literal equal to the operationId, which is a minor duplication smell but not a gap. GET/POST are often interchangeable per-method; picking one explicitly is fine. |
| **Models/types** | **GAP for polymorphic fields** | `chat.postMessage`'s `blocks`/`attachments` are heterogeneous arrays of Block-Kit objects discriminated by a `type` tag (JSON-Schema `oneOf`-style). ¬ß2's neutral type system (`string\|integer\|number\|boolean\|list<T>\|map<K,V>\|ref<Model>\|any`) has no discriminated-union primitive ‚Äî forces a fallback to `list<map<string,any>>`/`any`, losing the type fidelity refract otherwise promises. Also: `reactions.add` success is `{"ok":true}` with **zero other fields** ‚Äî leaner than ¬ß5's own `ack:` example which assumes an `ident` (e.g. `comment_id`) comes back; whether `Ack`'s `ident` is optional is unconfirmed by the frozen text ‚Äî a minor near-miss. |
| **Tests (beyond ReturnsError)** | **Degraded but not gapped** | `DrainsPages`/`RespectsLimit` are fine once/if the pagination near-miss is resolved. `RoundtripsBody`/`PropertyParse`/`PropertyRoundtrip` on `any`-typed Block Kit fields are weak-to-vacuous ‚Äî hypothesis can't meaningfully generate structured data for `any`. The upload orchestration (`getUploadURLExternal`‚Üíexternal PUT‚Üí`completeUploadExternal`) is explicitly out-of-scope per ¬ß10 ("Multi-step orchestrations... ‚Üí `Custom` handler + partial coverage") ‚Äî correctly anticipated by the spec, but means the auto-suite contributes essentially nothing there; coverage is 100% human-authored via `tests.add: [{strategy: Raw, ...}]`. |
| Async/streaming | **N/A / native** | All 11 sampled methods are synchronous req/response; no streaming surface hit in this sample (Slack does have RTM/Events API/Socket Mode, out of scope for Web API). |

## Verdict

refract's REST/status-based model fits Slack's *transport* (Bearer auth, sync HTTP, mostly-JSON
bodies, resource-shaped operationIds) surprisingly well ‚Äî the RPC-style all-POST path shape is
a **near-miss that resolves cleanly**, not a real gap, because Slack's dot-namespacing already
mirrors refract's directory layout. The genuine, spec-breaking mismatch is entirely on axis 9:
**Slack's 200-wrapped `ok:false` error envelope has no representation in a status-keyed
`responses:` map**, and this directly poisons the auto-generated `ReturnsError` test, which
would pass on a scenario Slack never produces while leaving the real failure path unverified.

## Top 3 gaps

1. **200-wrapped errors are a hard GAP.** refract needs a new error-strategy concept (e.g.
   `BodyFlag`: `{field: ok, error_field: error, false_value: false}`) that discriminates
   success/error models *within* a single HTTP status, not across statuses. Without it, the
   `responses:` map cannot express Slack's failure mode at all.
2. **`ReturnsError` test strategy is actively misleading for this API shape**, not just
   incomplete: it validates a 4xx/5xx path Slack doesn't use, and gives false confidence while
   the real `200 ok:false` path ‚Äî including the risk of misparsing an error body as a success
   model ‚Äî ships untested.
3. **Cursor pagination's `response_metadata.next_cursor` is a dotted path**, and the frozen
   spec never shows whether `Cursor`'s field config supports nesting (only a flat
   `items_field:` example exists) ‚Äî a near-miss that needs either an explicit dotted-path
   convention or is silently a GAP.

Secondary notes: Block Kit's polymorphic `blocks` array has no home in the neutral type system
(forced to `any`); 429 rate-limiting is a real non-2xx but cross-cutting with no shown
per-resource-default mechanism; form-encoded list fields need CSV serialization not covered by
`TypedModel`/`Assembled` as described.
