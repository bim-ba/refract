# refract stress test вЂ” Yandex 360 (Tracker + Wiki + Forms)

Source of truth: `references/yandex-360/{tracker,wiki,forms}/` (vendored docs) cross-checked
against the live SDK at `src/ycli/yandex/{tracker,wiki,forms}/**`. This is refract's home ground
(the generator was reportedly prototyped from this exact codebase), so the baseline expectation is
"maps cleanly" вЂ” the goal here is to find the near-misses anyway.

## Axis-by-axis verdict

| Axis | Verdict | One-line why |
|---|---|---|
| **auth** | **native** | Single `HeaderToken` (`Authorization: OAuth {token}` + `X-Org-Id: {organization_id}`) reused verbatim across all 3 domains' `_resource.yaml`. Real API optionally also accepts `Bearer <IAM-token>` + `X-Cloud-Org-ID` for Identity-Hub-bound orgs вЂ” ycli picked one canonical pair; that's a resource-owner decision, not a spec gap (a `Custom` or second `HeaderToken` entry would cover the alternate if ever needed). |
| **pagination** | **native for 3/5 shapes, GAP for 2** | `Offset` (Forms surveys), `Cursor` (Wiki descendants/grids), `RelativeCursor` (Tracker entities' `_relative` history/comments) all map 1:1 onto existing strategies. **GAP:** Tracker `issues._search` picks between 3 mechanics by *request shape* (plain page/`page`, `RelativeCursor`-via-response-`Link`-**header** for `queue`, and a scroll walk keyed off **response headers** `X-Scroll-Id`/`X-Scroll-Token` for >10k rows) вЂ” no built-in strategy reads a cursor from an HTTP header rather than the body; ycli itself doesn't auto-paginate this op (ships a single raw page + a manual `scroll_clear`). **Near-miss в†’ Custom:** Forms `answers.list_all` is nominally `NextUrl`, but the server hands back a *dead* relative `next_url` (a 404ing `/v3/...` route) that the client must detect and rewrite into a same-cursor query against the working `v1` route вЂ” `NextUrlStrategy`'s "pure mechanics, follow the URL" contract can't express a URL rewrite, so this is already hand-written `Custom` in ycli. |
| **request body** | **native for 4/5, GAP for 1** | `TypedModel` (Wiki `UploadSessionCreate`, Tracker `EntityCreate`/`BulkChangeUpdate`), `Assembled` (Tracker `issues create` CLI, which hand-assembles `{queue, summary, type:{key}, ...}` from scalar flags), `RawDict` (several `Custom`-adjacent raw bodies) all fit. **GAP:** Forms `questions.create`/`modify` take a **discriminated union of 12 typed models** (`StringQuestion \| BooleanQuestion \| ... \| SeriesQuestion`, tag=`type`, and `SeriesQuestion` is self-recursive) as ONE body вЂ” `TypedModel` in the spec names a single `model:`; there's no `oneOf`/union body strategy. Also a genuine binary-body mismatch: Wiki `upload_part` sends **raw bytes** under a forced `Content-Type: application/octet-stream` uplink `Body` (not JSON, not base64-in-JSON) вЂ” the registry's `Base64` strategy name implies a base64 string embedded in a JSON field, which is not what's happening on the wire; this either needs a distinct `RawBytes`/`Binary` strategy or a renamed/broadened `Base64`. |
| **responses & errors** | **native for the common case, 2 real near-misses** | Transport-level typed-error mapping (401/403в†’Auth, 404в†’NotFound, 429в†’RateLimit, 5xxв†’Server, elseв†’Client) matches the spec's "non-2xx в†’ typed error hierarchy" cleanly regardless of op. **Near-miss #1 (200-wrapped partial failure):** `GET /bulkchange/{id}/issues` returns `200` with a list where *individual items* carry `status: FAILED` + a structured `error` block вЂ” a partial-failure-in-200 envelope in the same family as the spec's flagged Slack `ok:false` case. **Near-miss #2 (ambiguous 200):** several Forms/Tracker `get`-style endpoints parse leniently вЂ” a 404 *or* an empty 2xx body both deserialize into an all-`None` model, so ycli layers a hand-written `require_found(sentinel=...)` guard in every such MCP tool; refract's `responses: {200: Model, 404: ErrorBody}` map assumes status code alone disambiguates, which it doesn't here. **Near-miss #3:** Forms `answers.export_results` вЂ” the SAME `GET` endpoint returns JSON (`200`/`202`, `{id,status,message}`) while running, but **302-redirects to a raw file** (non-JSON) once ready; `responses:` can't express "same status family, different content-type/schema depending on runtime state." |
| **operation/path shape** | **native** | Tracker's `entities` god-resource (`entity_type` = project\|portfolio\|goal as a *path parameter*, not a name-flattened resource) still fits refract's `<res>_<sub>_<verb>` operationId convention cleanly вЂ” the live MCP tool names (`tracker_entities_comments_create`, `tracker_entities_checklists_move`, `tracker_entities_set_permissions`) already follow exactly that pattern, with `entity_type` staying a runtime arg. `issues.move` (`POST /issues/{key}/_move?queue=`, no body, returns the full renamed `Issue`) and Wiki's DELETE-with-JSON-body (`grids.remove_rows`/`remove_columns`) are unusual HTTP shapes but both are just `method` + `path` + (optional) `body:` вЂ” no new spec concept needed. |
| **async/long-running** | **native for the poll loop, GAP for cross-resource wiring + multi-step upload** | The triggerв†’poll shape itself (`BulkChange.is_terminal`, `CloneOperationStatus.is_terminal`, generic `polling.poll(fetch, is_done, backoff, attempts)`) is uniform and would map onto one `AsyncOperation`-style strategy easily. **Near-miss:** the poll endpoint for Wiki clones lives on an *entirely different resource* (`wiki.operations.clone_get`/`gridclone_get`) than the trigger (`wiki.pages.clone`/`wiki.grids.clone`) вЂ” refract's per-operation spec has no field linking a trigger op to its cross-resource status op; that relationship is currently only documented in prose/docstrings. **Confirmed GAP (per spec В§10):** the 4-step upload pipeline (`uploadsessions.create` в†’ NГ— `upload_part` в†’ `finish` в†’ `attachments.attach`, the last call on yet another resource) cannot be one OpenAPI operation and is exactly the case В§10 already flags as `Custom handler` + partial coverage вЂ” validated here, not a new finding, but the home API confirms it's a *primary*, not edge-case, pattern (Wiki has it twice: pages and grids each have their own attach flow). |
| **models/types** | **native for scalars/refs, GAP for cross-field validators & polymorphic envelopes** | The neutral type system covers 90% of shapes (`EntityFields`, `Grid`, `Survey`, вЂ¦) including the `additionalProperties`-style escape (`EntityFieldsInput` uses `extra="allow"` for queue-local custom fields вЂ” JSON-Schema-aligned, so expressible via `any`/`map<string,any>` fallback). **Real GAP:** Forms `QuestionMove` carries a `model_validator(mode="after")` that *raises* client-side when `position` is set with no page/id/question target вЂ” compensating for a live API bug (bare `position` 200s but silently no-ops). The spec's model schema (В§2: `optional`/`enum`/`format`/`deprecated`) has no concept of a cross-field business-rule validator, and вЂ” more importantly вЂ” **no operation-level or model-level `handler:` escape hatch is defined for it**: `handler:` in В§5 is documented as "any surface" but the surfaces enumerated (auth/pagination/body/tests) don't include "model validation." This is a missing refract concept, not just a missing strategy. |
| **tests** | **native and non-vacuous for the common ops; would need care for 3 of the 14** | `ParsesModel`/`ReturnsError`/`RespectsLimit`/`WrapsAck` are all real, meaningful checks for ops 1, 2 (single mechanic), 7, 8, 9, 12 вЂ” ycli's actual test layout (`test_client.py`/`test_cli.py`/`test_mcp.py`/`test_models.py` per resource, `responses`-stubbed) is close to the spec's auto-suite in spirit. **Would be vacuous/wrong as specified:** `DrainsPages`/`RespectsLimit` against `issues.search` вЂ” there's no single pagination mechanic to drain (see pagination axis), so a generated test would need to pick one of 3 request shapes and could silently test the wrong one. `RoundtripsBody` against `questions.create`/`modify` would need 12 separate fixtures (one per union member) or it only ever exercises whichever member `faker: Question` happens to synthesize вЂ” a naive `faker`-seeded single-model roundtrip test under-covers a discriminated union. `WrapsAck` for `bulk_update`/`entities.bulk_update` is misleading if applied naively вЂ” the op returns a `BulkChangeOperation` handle, not an immediate `Ack`; the *actual* ack-shaped moment is several polls later, which the Tier-1 `responses`-stubbed suite would need to fake via injected `sleep`/`backoff`, matching ycli's own `polling.poll(sleep=..., backoff=...)` injection pattern вЂ” doable, but only if the test-strategy registry explicitly supports polling injection (not mentioned in В§8). |

## Top gaps / near-misses (loudest first)

1. **Header-driven pagination cursors (GAP).** Tracker issue search's `queue`-scoped relative pagination reads its next-page id from a response **`Link` header**, and its scroll mode reads `X-Scroll-Id`/`X-Scroll-Token` from response **headers**, feeding them back as query params. Every built-in pagination strategy in В§6 reads `next_of`/`items_field` off the parsed **body** model. None can read a header. This is the same class of gap the spec already flags for GitHub's Link-header pagination вЂ” confirmed here on refract's own home API, and worse (mixed header+query, two independent header-driven mechanics on one endpoint, request-shape-selected).

2. **No model-level (cross-field) validator concept, and no escape hatch for one.** `QuestionMove`'s `model_validator` exists purely to prevent silently no-op-ing against a real API bug. The spec's neutral type system (В§2) has no field-level-rule vocabulary beyond `optional`/`enum`/`format`/`deprecated`, and `handler:` (В§5) is scoped to op-level surfaces, not model validation. A generator that only emits В§2-shaped models would ship a client that reproduces the silent-no-op bug.

3. **Polymorphic/union request bodies.** `TypedModel` (В§7) names one `model:`; Forms questions need one of 12 (recursively, for `series`). Either the body registry needs a `OneOf`/discriminated-union body strategy, or every such op degrades to `Custom` вЂ” which then also drags the auto-test-suite's `RoundtripsBody`/`PropertyRoundtrip` down to single-member coverage unless the spec can express "faker across every union member."

4. (Runner-up) **Trigger/poll operations that live on different resources** (Wiki clone) and **200-body-encodes-failure** (Tracker bulk-issues, Forms lenient-404) both work today only because ycli hand-writes the linking logic and the sentinel guards вЂ” neither has a first-class spec field.
