## Per-tool detail

**datamodel-code-generator.** Confirmed alive and the most current tool surveyed: v0.68.1 shipped 2026-07-08, six days before this report. Default output is pydantic v2 `BaseModel`; pydantic v1 output was recently *removed* (v1 runtime compat layer dropped), reinforcing it's tracking pydantic v2 as the sole path. Custom Jinja2 templates are documented and directory-scoped by output model type (`custom_templates/pydantic_v2/BaseModel.jinja2`), confirmed via Context7 (`/koxudaxi/datamodel-code-generator`, retrieved 2026-07-14). This fully re-validates the prior session's conclusion that it's the one reusable in-repo component. Sources: [GitHub](https://github.com/koxudaxi/datamodel-code-generator), [PyPI](https://pypi.org/project/datamodel-code-generator/), [Releases](https://github.com/koxudaxi/datamodel-code-generator/releases), Context7 `/koxudaxi/datamodel-code-generator`.

**openapi-generator.** Very active (monthly-plus releases per DeepWiki), and confirmed via DeepWiki (`OpenAPITools/openapi-generator`, retrieved 2026-07-14) to support server stubs (`python-fastapi`), docs, and schema outputs beyond a plain SDK, all Mustache/Handlebars-templatable. However the Python-specific pydantic v2 gap is real and current: GitHub issue #16468 "[PYTHON] update to pydantic v2" is still open, and `python-pydantic-v1` is exactly what its name says. No generator here targets a CLI or an MCP server. Verdict unchanged from prior session: not a fit for the models layer, and not a multi-target tool. Sources: [DeepWiki Q&A](https://deepwiki.com/search/can-openapigenerator-emit-anyt_05579662-6a75-422f-ab99-7a53827204f1), [issue #16468](https://github.com/OpenAPITools/openapi-generator/issues/16468), [openapi-generator.tech/docs/templating](https://openapi-generator.tech/docs/templating/).

**Swagger Codegen.** Still exists, still gets community contributions, but is not actively developed by the original Swagger team and lags OpenAPI Generator's release cadence by a widening margin. No reason to pick it over openapi-generator in 2026. Source: WebSearch synthesis citing [loquisoft.com comparison](https://loquisoft.com/blog/swagger-codegen-vs-openapi-generator-which-is-more-active/) and [openapi-generator.tech migration guide](https://openapi-generator.tech/docs/swagger-codegen-migration/), retrieved 2026-07-14.

**TypeSpec.** Reached 1.0 GA for its core spec-authoring layer (compiler, `@typespec/http`, `@typespec/openapi3`, `@typespec/json-schema` all stable) — but the Python client emitter (`@typespec/http-client-python`) remains explicitly **preview**, with open issues in mid-2026 about basic gaps (e.g. streaming binary body typing, #11107). No server/CLI/MCP emitter surfaced. Adopting TypeSpec would also mean re-authoring the ~50 endpoint specs in a new DSL rather than a lightweight concise spec of our own design. Not ready. Sources: [typespec.io 1.0-RC blog](https://typespec.io/blog/typespec-1-0-RC-release/), [1.0.0 release notes](https://typespec.io/release-notes/typespec-1-0-0/), [microsoft/typespec issues](https://github.com/microsoft/typespec/issues), Context7 `/microsoft/typespec`, retrieved 2026-07-14.

**Smithy / smithy-python.** The `smithy-lang/smithy-python` GitHub repo is not indexed on DeepWiki (query returned "Repository not found"), so this is corroborated by WebSearch only: the repo's own README describes it as "in development," and only the `restJson1` protocol is supported. It's real evidence that Smithy is protocol- not AWS-coupled (confirmed usable outside AWS per AWS's own blog and Smithy docs), but the Python codegen story is immature and there's no CLI/MCP output. Given the extra IDL + JVM-toolchain investment for one incomplete Python target, not worth pursuing now. Sources: [smithy-lang/smithy-python](https://github.com/smithy-lang/smithy-python), [AWS Introducing Smithy for Python blog](https://aws.amazon.com/blogs/developer/introducing-smithy-for-python/), [smithy.io](https://smithy.io/2.0/), retrieved 2026-07-14. **COULDN'T-VERIFY**: exact current release maturity/version number (DeepWiki index unavailable; would need a direct repo read to pin down further).

**Fern.** Confirmed **acquired by Postman on 2026-01-08** (Postman's own blog, BusinessWire, TechCrunch-adjacent coverage) with the stated intent to keep "product, brand, and roadmap unchanged." This resolves ambiguity from an initial search result that vaguely said "acquired by Postman" without a date. Since the prior 2026-07-10 rejection, Fern shipped a **CLI generator** (`fernapi/fern-cli`) whose single binary doubles as an MCP server over stdio/streamable-HTTP — genuinely new multi-target capability that undercuts the "SDK-only" part of the original rejection. DeepWiki confirms (`fern-api/fern`, retrieved 2026-07-14) Fern still routes everything through its own Intermediate Representation before any generator runs — i.e. the "foreign model identity" objection is unchanged — and generation is still Docker-image-per-language via `generators.yml`, i.e. still an external pipeline you plug your spec into rather than a component you embed. `fern generate --local` does run without a SaaS login/Docker-only, addressing part of the cost/lock-in concern. Net: worth a narrow, time-boxed spike of `fern-cli` specifically for "one spec → CLI + MCP" if that combination becomes a real priority, but it still isn't a fit for generating pydantic-v2 models inside our existing package layout, and the prior rejection's core architectural objections (foreign IR, external pipeline ownership) still stand. Sources: [Postman blog](https://blog.postman.com/postman-acquires-fern/), [Fern CLI generator post](https://buildwithfern.com/post/cli-generator), [Fern self-hosted docs](https://buildwithfern.com/learn/sdks/deep-dives/self-hosted), DeepWiki `fern-api/fern`.

**Stainless.** Prior session's "sunsetting" claim is now **fully confirmed with a hard date**: Anthropic acquired Stainless on 2026-05-18 for a reported >$300M and told TechCrunch it will wind down all hosted Stainless products, including the central SDK/MCP generator; existing customers retain already-generated code but no new generation capability continues. This is a load-bearing update worth flagging prominently: it's now definitively dead, not merely rumored. Sources: [TechCrunch](https://techcrunch.com/2026/05/18/anthropic-has-acquired-the-dev-tools-startup-used-by-openai-google-and-cloudflare/), [The New Stack](https://thenewstack.io/anthropic-stainless-sdk-acquisition/), [WinBuzzer](https://winbuzzer.com/2026/05/19/anthropic-buys-stainless-ends-hosted-sdk-tools-xcxwbn/), retrieved 2026-07-14.

**Speakeasy.** Actively developed, ships a real "standalone MCP server generation" product, and supports Python for plain SDK generation. But as of this check, MCP server generation is explicitly **TypeScript-only** ("additional language support coming soon" — no committed date), so it can't produce our Python MCP layer today. Pricing is commercial SaaS (free tier capped at one SDK / ~50 methods; 14-day paid trial then reverts to free). Not actionable now; worth re-checking if/when Python MCP output ships. Sources: [speakeasy.com/product/mcp-server](https://www.speakeasy.com/product/mcp-server), [speakeasy.com/blog/generate-mcp-from-openapi](https://www.speakeasy.com/blog/generate-mcp-from-openapi), retrieved 2026-07-14.

**New entrants (2025–2026).** Two are worth naming distinctly from the noisier field of SaaS "OpenAPI-to-MCP-in-5-minutes" products (DigitalAPI, Zuplo, Gram, awslabs openapi-mcp-server, harsha-iiiv/openapi-mcp-generator — the last is TypeScript-only, generates a standalone MCP server, not Python):
- **`cnoe-io/openapi-mcp-codegen`** — confirmed via GitHub API (retrieved 2026-07-14): Apache-2.0, 39 stars, 9 forks, 6 open issues, last push 2026-05-19, not archived. It's a small community project (CNOE.io/CAIPE, CNCF-adjacent), not a single-vendor product, and it is the only surveyed tool that is simultaneously **Python-native, Jinja2-templated, local/free, and emits an MCP server directly** (with tools + models + client code) from an OpenAPI spec. This is new information not available to the prior session and is the strongest lead for "generate the MCP layer (and maybe models) from spec" — but it's unproven at our scale and needs a hands-on spike, not just a docs read, before any verdict beyond "promising, unverified."
- **`FastMCP.from_openapi()`** — not a new tool but a capability that matured materially since FastMCP's "Route Awakening" (v2.5.0): route-map-based control over which OpenAPI operations become MCP tools/resources, tag-based mapping, and renaming, confirmed via Context7 (`/websites/gofastmcp`, retrieved 2026-07-14). Since FastMCP is **already a ycli dependency**, this is the lowest-friction thing to prototype: point it at a domain's OpenAPI/JSON-Schema spec and see how close the auto-generated MCP surface gets to ycli's hand-written `mcp.py` files and ARCH-3 annotation-honesty requirements. It only touches the MCP layer, not CLI/models/tests, and by default it mounts a live server rather than emitting committed, reviewable source — that gap would need to be closed (e.g. exporting/materializing generated tool defs to files) before it fits ycli's "generate + commit + review" model.

## Does the prior session's conclusion still hold?

Mostly yes, with two updates and one hardened fact:

1. **Hardened, not just held**: "Stainless is sunsetting" is now a confirmed, dated fact (Anthropic acquisition, 2026-05-18, hosted generator winding down) — stronger evidence than the prior session had.
2. **datamodel-code-generator remains the one clearly reusable in-repo component** — re-confirmed as actively maintained (release six days before this report) with first-class pydantic v2 + custom-template support.
3. **Fern's rejection needs a footnote, not a reversal**: it now generates a CLI+MCP binary alongside the SDK (closing the "SDK-only" gap), but the "foreign model identity" (Fern IR) and "inverted ownership" (external Docker-based generator pipeline) objections are unchanged per DeepWiki's read of the current codebase.
4. **The genuinely new fact the prior session missed**: no dedicated "YAML spec → SDK+CLI+MCP+tests" tool exists in 2026 either, but the MCP corner of that problem now has real, free, Python-native options that didn't get surfaced before — `cnoe-io/openapi-mcp-codegen` and, more relevantly since it's already a dependency, `FastMCP.from_openapi()`. Both deserve a hands-on spike focused narrowly on the MCP layer before the "42-endpoint backlog" roadmap locks in a fully hand-authored MCP tier.

**COULDN'T-VERIFY items**: Smithy-python's precise current release/version number (DeepWiki has no index for the repo); Speakeasy's exact committed timeline for Python MCP support (described only as "coming soon" in their own materials); Fern's exact current pricing tiers for 2026 (search returned only tier names, not up-to-date numbers).
